# Replication

Volume replication is asynchronous and primary-to-standby. It runs in two phases: bulk segment transfer copies
existing data to bring a standby current, then streaming change data capture applies incremental mutations as they
happen.

The changelog drives streaming replication. It is an ordered log of all volume mutations, stored in FoundationDB.
All replication state is persisted in FoundationDB, so transfers resume across restarts.

## Changelog

### Purpose and Design

The changelog records every volume mutation (APPEND, INSERT, UPDATE, DELETE) in temporal order. It lives in
FoundationDB under the volume's directory subspace and is the source for streaming replication.

Each entry gets a sequence number from a Hybrid Logical Clock (HLC), which gives a total order across all operations
on a volume.

The changelog uses three FoundationDB subspaces:

| Subspace                          | Byte | Purpose                                             |
|-----------------------------------|------|-----------------------------------------------------|
| `CHANGELOG_SUBSPACE`              | 0x05 | Forward changelog. Stores mutation entries          |
| `MUTATION_TRIGGER`                | 0x06 | Watch key. Notifies standbys of new entries         |
| `CHANGELOG_BACK_POINTER_SUBSPACE` | 0x07 | Reverse index. Segment position to sequence numbers |

### Hybrid Logical Clock

Sequence numbers are generated by a `HybridLogicalClock` that produces 64-bit timestamps:

```
| 44-bit physical time (ms) | 20-bit logical counter |
```

Properties:

- **Monotonically increasing**: physical time ratchets forward; the logical counter handles same-millisecond events
- **Throughput**: about 1 million ops/ms before the logical counter overflows, at which point it spin-waits for the
  next millisecond
- **Clock drift tolerance**: up to 5 seconds of backward system clock drift; beyond that it throws
- **Total order**: sequence numbers are globally ordered within a volume

The `next(physicalNow)` method is synchronized. It compares the incoming wall-clock time against the last physical
time, advances the logical counter for same-millisecond events, and packs the result as
`(lastPhysical << 20) | lastLogical`.

### Entry Structure

**Key:**

```
(CHANGELOG_SUBSPACE, sequenceNumber, parentOperationKind, operationKind, versionstamp)
```

**Value** (tuple-packed):

- For APPEND, INSERT, DELETE: `(segmentId, position, length, prefix)`
- For UPDATE: `(segmentId, position, length, prevSegmentId, prevPosition, prevLength, prefix)`

**ParentOperationKind** categorizes mutations by replication behavior:

| Kind           | Byte | Operations             | Meaning                                 |
|----------------|------|------------------------|-----------------------------------------|
| `LIFECYCLE`    | 0x01 | APPEND, INSERT, UPDATE | Operations that create or relocate data |
| `FINALIZATION` | 0x02 | DELETE                 | Operations that remove data             |

**OperationKind** identifies the specific mutation:

| Kind   | Byte |
|--------|------|
| APPEND | 0x01 |
| DELETE | 0x02 |
| UPDATE | 0x03 |
| INSERT | 0x04 |

**Write behavior** varies by operation:

- **APPEND** uses `SET_VERSIONSTAMPED_KEY` with an incomplete versionstamp, assigned at commit time
- **INSERT**, **UPDATE**, **DELETE** use `set()` with a complete versionstamp that is already known

Two supporting records carry decoded changelog data:

- `ChangeLogCoordinate(sequenceNumber, segmentId, position, length)` is a single segment location
- `ChangeLogEntry(kind, versionstamp, before, after, prefix)` is a full decoded entry, where `before` and `after`
  are optional `ChangeLogCoordinate` values depending on the operation kind:
    - APPEND/INSERT: only `after` is present
    - UPDATE: both `before` (old location) and `after` (new location) are present
    - DELETE: only `before` is present

### Back-Pointer Index

The back-pointer index supports reverse lookups: given a segment position, find the changelog sequence number(s) for
it.

**Key:**

```
(CHANGELOG_BACK_POINTER_SUBSPACE, segmentId, position, sequenceNumber)
```

**Value:** empty bytes

A back pointer is written for every changelog entry. UPDATE operations create two back pointers, one for the old
position and one for the new position.

The main consumer is `resolveTailSequenceNumber`, which uses the back-pointer index to find the changelog sequence
number that corresponds to a segment's tail pointer. This runs during the handoff from segment replication to change
data capture.

### Safe Watermark

The safe watermark tracks which sequence numbers are known committed, so replication never reads uncommitted
changelog entries.

**Mechanism:**

1. When a sequence number is generated, it is added to a `ConcurrentSkipListSet` of in-flight sequence numbers
2. A completion callback is registered via `tr.getVersionstamp()`. When the transaction commits, the callback
   removes the sequence number from the in-flight set, with a 10-second timeout safety net for rolled-back
   transactions
3. The watermark is computed as:
    - If the in-flight set is non-empty: `min(in-flight) − 1`
    - If the in-flight set is empty: `lastGeneratedSequenceNumber`

**Concurrency:**

The watermark computation uses a `StampedLock` with optimistic reads for the fast path. If the optimistic read fails
validation (concurrent write), it falls back to a regular read lock. Write mutations (removing from the in-flight set)
acquire the write lock.

**Consumer:**

The `CHANGELOG.WATCH` handler returns `min(safeWatermark, latestSequenceNumber)`, so standbys never see sequence
numbers from uncommitted transactions.

### Retention and Pruning

Changelog entries accumulate over time. Pruning removes entries older than a configured retention period.

- `calculateCutoffEnd(context, retentionPeriod)` converts a retention period (in hours) to an HLC cutoff timestamp
  using `HybridLogicalClock.fromPhysicalTime()`
- `prune(tr, cutoffStart, cutoffEnd, segmentMaxPositions)` clears forward changelog entries and their corresponding
  back pointers within the specified range

This bounds changelog growth while keeping enough history for active standby nodes to catch up.

## Replication

### Architecture Overview

Replication is per-shard (`ShardKind` x `shardId`), with each shard owning its own volume. Only standby nodes run
replication; they pull data from the primary. Communication uses internal RESP commands over Lettuce connections to
the primary's internal port.

All replication state is persisted in a per-standby FoundationDB directory subspace:

```
kronotop/{cluster}/metadata/volumes/bucket/volume/{volumeName}/standby/{standbyId}
```

**Component hierarchy:**

```
ReplicationService (lifecycle manager, one per node)
  └─ ReplicationWatchDog (per shard, retry supervisor)
       └─ VolumeReplication (orchestrator)
            ├─ SegmentReplication (Phase 1: bulk transfer)
            └─ ChangeDataCapture (Phase 2: streaming)
```

### Internal Replication Protocol

Four internal RESP commands handle data transfer. The primary serves these through `VolumeService` handlers:

| Command               | Handler                     | Purpose                                                                                     |
|-----------------------|-----------------------------|---------------------------------------------------------------------------------------------|
| `SEGMENT.TAILPOINTER` | `SegmentTailPointerHandler` | Returns `(nextPosition, sequenceNumber)` for a segment, the upper bound for bulk transfer   |
| `SEGMENT.RANGE`       | `SegmentRangeHandler`       | Returns raw byte chunks from a segment at specified `(position, length)` ranges             |
| `CHANGELOG.WATCH`     | `ChangeLogWatchHandler`     | Blocks until committed entries exist beyond a given sequence number; returns safe watermark |
| `CHANGELOG.RANGE`     | `ChangeLogRangeHandler`     | Returns changelog entries within a sequence number range, filtered by `ParentOperationKind` |

### Replication State

Per-segment replication state is stored in FoundationDB via the `ReplicationState` class:

| Field           | Key suffix | Type         | Description                                            |
|-----------------|------------|--------------|--------------------------------------------------------|
| Position        | 0x01       | long         | Current byte offset within the segment                 |
| Error message   | 0x02       | String       | Last failure root cause                                |
| Status          | 0x03       | enum         | WAITING → RUNNING → DONE / FAILED / STOPPED            |
| Tail pointer    | 0x04       | (long, long) | `(sequenceNumber, nextPosition)` at segment completion |
| Stage           | 0x05       | enum         | SEGMENT_REPLICATION or CHANGE_DATA_CAPTURE             |
| Sequence number | 0x06       | long         | Last applied changelog sequence number                 |

Key structure: `(REPLICATION_STATE=0x00, fieldType, segmentId)` within the standby's subspace.

The cursor (`readCursor`) reads the highest segment ID with a stored position; that fixes where replication resumes
after a restart.

### Phase 1: Segment Replication

Segment replication transfers existing segment data from the primary in configurable chunks.

**Algorithm:**

1. Fetch tail pointer via `SEGMENT.TAILPOINTER(volume, segmentId)` → `(limitPosition, sequenceNumber)`
2. Store tail pointer in the replication state
3. Loop while `position < limitPosition`:
    - Calculate chunk length: `min(chunkSize, segmentSize − position)`
    - Fetch chunk via `SEGMENT.RANGE(volume, segmentId, [(position, length)])`
    - Write chunk to a local segment file via `RandomAccessFile` + NIO `FileChannel`
    - Sync file descriptor
    - Persist position, clamped to `min(position, limitPosition)` to avoid overshooting active segments
4. When `position ≥ limitPosition`: mark segment status as DONE

**Resumability:** on restart, the cursor reads the persisted position and continues from where it left off.

**Position clamping** matters for active segments where the primary is still writing. Batch writes may overshoot the
tail pointer boundary, so the persisted position is always clamped to the lower of `(position, limitPosition)`.

### Phase 2: Change Data Capture

Change data capture (CDC) streams incremental changes from the changelog after segment replication completes.

**Cursor initialization:**

- If transitioning from segment replication: use the tail pointer's sequence number
- If resuming an existing CDC session: use the stored sequence number

**Algorithm:**

1. Loop until shutdown:
    - **Watch:** `CHANGELOG.WATCH(volume, sequenceNumber)` blocks until new committed entries exist; returns
      `latestSequenceNumber`
    - **Fetch:** `CHANGELOG.RANGE(volume, LIFECYCLE, "(seqNum", "latestSeqNum]", null)` returns a list of changelog
      entries
    - **Process changes in batches:**
        - Accumulate `SegmentRange` entries up to `chunkSize`
        - Detect segment rollover (entry's `segmentId ≠ current segmentId`)
        - Fetch data via `SEGMENT.RANGE(volume, segmentId, ranges)`
        - Write to a local segment file, sync
        - Persist progress (`sequenceNumber`, `position`)
    - **On segment rollover:** close the old file, open a new segment file, update checkpoint

CDC only queries `LIFECYCLE` operations (APPEND, INSERT, UPDATE). DELETE operations are excluded because they produce
no new data to transfer; deletes only remove metadata in FoundationDB.

CDC uses a separate `ReplicationClient` without timeouts for the blocking `CHANGELOG.WATCH` call.

### VolumeReplication Orchestrator

`VolumeReplication` is the main loop that connects to the primary and sequences the two phases.

**`resolveNextSegment()`** determines the next action based on cursor state:

| Status           | Action                                                                                        |
|------------------|-----------------------------------------------------------------------------------------------|
| WAITING, RUNNING | Continue current segment; switch to CDC if stage is CHANGE_DATA_CAPTURE                       |
| DONE             | Find next segment via binary search on the primary's segment list; if none exists → start CDC |
| FAILED, STOPPED  | Stop replication                                                                              |

Each iteration creates either a `SegmentReplication` or `ChangeDataCapture` stage. The stage runs to completion or
shutdown, then the loop iterates to pick the next step.

### ReplicationWatchDog

`ReplicationWatchDog` wraps `VolumeReplication` with retry logic running in an executor thread.

**Retry behavior:**

- On failure: increment retry counter
- If elapsed time since the last failure exceeds the reset threshold: reset counter to 0
- If retry count exceeds `maxRetries`: permanently stop replication for that shard
- Before each attempt: verify route membership (stop if this node is no longer a standby)

**Configuration:**

| Property                             | Default | Description                             |
|--------------------------------------|---------|-----------------------------------------|
| `volume.replication.max_retries`     | 10      | Maximum consecutive retry attempts      |
| `volume.replication.retry_interval`  | 10s     | Wait time between retries               |
| `volume.replication.reset_threshold` | 10s     | Elapsed time to reset the retry counter |

### ReplicationService

`ReplicationService` is the lifecycle manager, one per node, that manages all replication watchdogs across shards.

**Startup:** iterates all shard IDs, starts replication for shards where this node is a standby.

**Thread pool:** cached executor with named threads (`kr.volume-replication-%d`).

**Suppression flag:** prevents auto-restart after an explicit stop. Cleared only by an explicit start. This stops
routing-event-triggered restarts from overriding a deliberate stop.

**Routing event hooks:**

| Event                   | Behavior                                                                      |
|-------------------------|-------------------------------------------------------------------------------|
| `START_REPLICATION`     | Start replication (respects suppression unless the call is explicit)          |
| `STOP_REPLICATION`      | Stop the replication watchdog                                                 |
| `PRIMARY_OWNER_CHANGED` | If this node became primary: stop. If still standby: reconnect to new primary |

### ReplicationClient

`ReplicationClient` manages a stateful Lettuce connection to the primary node.

**Connection design:**

- **Lock-free fast path** for `conn()` via volatile reads, with no lock acquisition in the common case
- **Lock-protected state mutations** for `connect()` and `reconnect()` via `ReentrantLock`
- **Connection routing** uses `RoutingService` to find the primary's internal address

**Connection establishment:**

- 3-retry loop with 1-second waits for transient failures (ConnectException, SocketTimeoutException,
  UnknownHostException, RedisConnectionException, RedisCommandTimeoutException)
- If already connected to the same server, returns immediately (Lettuce handles network blips via auto-reconnect)
- On topology change: closes the existing connection, creates a new client to the new primary

**Not-connected backoff:** During a topology-change reconnect, `connected` is briefly false. A stage that calls
`conn()` in this window receives `ReplicationClientNotConnectedException`, aborts without marking the segment failed,
and the `VolumeReplication` loop re-dispatches it. Before returning, the stage waits
`volume.replication.reconnect_backoff` (default 250ms) so the loop does not busy-spin while the new connection is
being established.

**CDC timeout handling:** CDC uses a second `ReplicationClient` instance without timeouts for the blocking
`CHANGELOG.WATCH` operation, which can block indefinitely waiting for new mutations.

### Consistency Guarantees

| Guarantee                | Mechanism                                                                                                                                         |
|--------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------|
| **No data loss**         | Segment replication copies up to the tail pointer; CDC starts from the tail's sequence number                                                     |
| **No uncommitted reads** | `CHANGELOG.WATCH` returns `min(safeWatermark, latestSequenceNumber)`                                                                              |
| **Position clamping**    | Segment replication never advances the persisted position past `limitPosition`                                                                    |
| **Resumability**         | All progress is persisted in FoundationDB; restart picks up from the exact position                                                               |
| **Metadata validation**  | `resolveTailSequenceNumber` validates versionstamp alignment via the back-pointer index; throws `InconsistentVolumeMetadataException` on mismatch |
